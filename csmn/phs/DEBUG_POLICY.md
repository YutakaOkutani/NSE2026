# csmn/phs 設計・デバッグ指針

`phs/` はミッションの意思決定層である。ここで最も大事なのは、各フェーズが「何を達成したら次へ進むのか」「何を危険とみなして戻るのか」「いつ諦めるのか」を明文化することである。

## この層の完成条件

- 各フェーズの目的が1文で説明できる
- 入口条件、維持条件、脱出条件が区別されている
- フェーズ遷移の理由がログや出力から分かる
- 誤検知やセンサー欠落に対する保守動作がある

## 各フェーズに必ず持たせる視点

- 目的
  - そのフェーズで達成したいことは何か
- 成功条件
  - 何が観測されたら次へ進むのか
- 失敗条件
  - 何が起きたら戻る、諦める、停止するのか
- 観測根拠
  - その判断に使ったセンサーと信頼条件は何か
- 時間戦略
  - なぜそのタイムアウトが必要か

## フェーズごとの設計思想

- `p0`
  - 降下・着地の誤判定を避けながら、遷移機会を逃しすぎない
- `p1`
  - 展開動作は安全優先で、再現性を損なう近道を入れない
- `p2`
  - キャリブレーションは以後の航法品質を左右する前提整備
- `p3`
  - 遠距離移動フェーズ。方位ソースの信頼性を意識した保守的判断が必要
- `p4`
  - 発見フェーズ。誤検知を抑えつつ、探索不能状態を長引かせない
- `p5`
  - 接近フェーズ。見失いと到達誤判定の両方を制御する
- `p6`
  - 最終突入。短時間・高確度・安全停止への接続を重視する
- `p7`
  - 終了状態。停止・通知・ログ確定が仕事

## 改善指針

- フェーズ判断は「その場の値」だけでなく、「連続性」「鮮度」「確認回数」を使う
- 1つのフェーズが複数役割を持ちすぎたら分離を検討する
- 戻り遷移は恥ではなく、成功率を上げる戦略として設計する
- timeout は失敗ではなく、次善策へ移るための戦略的判断として定義する

## デバッグで重視すること

- なぜ遷移したか
- なぜ遷移しなかったか
- どのセンサーを信用したか
- timeout や fallback が想定どおり効いたか

## 避けたい実装

- 閾値1本で全状況を説明しようとすること
- 入口時の一時状態リセットが曖昧なこと
- 遷移理由の説明不能な phase update
