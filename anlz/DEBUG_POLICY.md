# anlz 設計・デバッグ指針

`anlz/` は実験結果を次の改善へつなげるための解析層である。ここでは可視化そのものが目的ではなく、実験の失敗や成功を設計改善へ変換できることを価値とする。

この層には大きく 2 つの役割がある。

- `log.py`: ゴール確度を上げるためのデバッグ解析。フェーズ、センサー異常、航法、視覚検出、モータ指令、timeout / fallback を追い、次の制御改善へつなげる。
- `explorer.py`: CanSat を本来の模擬探査機として扱う探査再構成解析。GPS、高度、障害物、視覚検出、必要に応じてカメラ画像を使い、走行した地形・経路・周辺環境を再現する。

## 目的

- ログから挙動を説明できるようにする
- 異常の兆候を早く見つける
- 閾値調整や戦略変更の根拠を得る
- 模擬探査機として、走破した地形・障害物・目標接近の様子を再構成する
- センサー群を「ゴールするため」だけでなく「環境を観測するため」のデータとして活用する

## この層に求める完成度

- ログ欠損や列追加に比較的強い
- 重すぎず、実験後にすぐ回せる
- どの phase で何が起きたか追いやすい
- 改善議論に使える観点が揃っている
- デバッグ解析と探査再構成解析の出力先が混ざらない

## 出力先の分離

- `log.py` の出力は `anlz/outputs/<log名>/run_.../` に置く
- `explorer.py` の出力は `anlz/explorer_outputs/<log名>/run_.../` に置く
- 解析目的が違うため、ファイル名やディレクトリを無理に共通化しない
- ただし、同じログ列の読み方や品質判定は可能な限り共有し、解釈のズレを減らす

## 重視する観点

### デバッグ解析

- フェーズごとの滞在時間
- センサー stale と判断の相関
- GPS 品質と航法挙動
- 視覚確率の時間変化
- モータ指令と終端理由
- timeout / fallback の発生箇所

### 探査再構成解析

- `LAT` / `LNG` からローカル座標系へ変換した走行経路
- `ALT` による相対高度・地形プロファイル
- `ObstacleDist` と進行方向から推定した障害物候補点
- `ConeDir` / `ConeProb` / `ConeMethod` による目標・物体検出イベント
- `Phase` と地形・障害物・視覚イベントの位置関係
- `GPSFixQual` / `GPSSats` / `GPSHdop` による再構成結果の信頼度
- カメラ画像がある場合は、ログ時刻に近い画像とイベントを対応づける

## 改善指針

- 新しいログ列が増えたら、解析価値の高いものから追従する
- 異常判定は現場の感覚ではなく、運用ルールに近づける
- 「グラフがある」だけで終わらず、次の修正仮説へつながる出力を意識する
- `explorer.py` は、制御の反省だけでなく「探査成果物」として見られる出力を意識する
- 地形再構成は実測点の密度・GPS 品質・高度センサーの信頼度に強く依存するため、過度に精密な地図として扱わず、信頼度が分かる形で出す
- 障害物点は現状では距離センサーと進行方向からの推定であり、実物位置の確定値ではないことを出力・解釈で明確にする
- カメラ活用は、まずイベント時刻との画像対応づけを優先し、将来的に特徴点・深度推定・3D 再構成へ拡張する

## 避けたい状態

- 実験後に重すぎて回されない
- 出力が多いだけで判断に使えない
- 本番ログ仕様変更に追従できない
- デバッグ目的と探査再構成目的の出力が混ざり、どちらの判断材料か分からなくなる
- きれいな 3D 表示だけが先行し、GPS 品質や高度誤差を無視した過信につながる
