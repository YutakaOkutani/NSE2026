# runs/spec 設計・デバッグ指針

`spec/` は仕様を固定し、ロジックの退行を防ぐ場所である。実機試験の価値は高いが、境界条件を毎回現場でしか確認できない状態は完成度が低い。ここでは、ハード無しで守れる品質を最大化する。

## 目的

- 判定ロジックの境界を固定する
- 過去不具合の再発を防ぐ
- 実機依存の少ない改善を支える

## 優先対象

- フェーズ遷移条件
- タイムアウト境界
- センサー無効時の挙動
- 誤検知抑制条件
- fallback 条件

## 指針

- 小さなフェイクで本質を再現する
- 条件の意味が読み取れるテスト名にする
- バグ修正時は可能なら先に失敗ケースを追加する
- 実機不具合でも、判断部だけは spec 化できないか考える

## このフォルダが強くなると得られること

- 実験前の安心感が増す
- 閾値や条件変更の副作用を早く検知できる
- 現場での場当たり修正が減る
